## ADR-006: API 서비스 레이어 추상화 (Dacon API Service)

- **날짜:** 2024-08-05
- **상태:** 채택됨

### 컨텍스트 (Context)
- 기존에는 `App.tsx` 내의 `fetchAllData` 함수에서 직접 `fetch`를 사용하여 여러 API를 호출했습니다.
- 프로젝트가 확장됨에 따라 API 엔드포인트가 늘어났고(v1, v2 혼용), 호출 로직이 UI 컴포넌트와 강하게 결합되어 가독성과 유지보수성이 저하되었습니다.
- 검색 정렬 로직 등에서 API로부터 받은 데이터의 형식을 전처리해야 할 필요성이 생겼습니다.

### 결정 (Decision)
- 모든 외부 API 호출을 전담하는 **`services/daconApi.ts`** 모듈을 생성하여 데이터 페칭 로직을 추상화하기로 결정했습니다.

### 근거 (Rationale)
- **관심사의 분리 (Separation of Concerns):** UI 로직(React 컴포넌트)과 데이터 취득 로직(API 호출)을 분리하여 각 코드의 역할을 명확히 합니다.
- **코드 재사용성:** 동일한 API를 다른 컴포넌트나 유틸리티에서 필요로 할 때 중복 코드 없이 간단히 호출할 수 있습니다.
- **중앙 집중식 관리:** API Base URL이나 인증 방식이 변경될 경우, 여러 파일을 수정할 필요 없이 서비스 모듈 한 곳만 수정하면 됩니다.
- **에러 핸들링 통합:** 모든 요청에 대한 에러 처리를 서비스 레이어에서 일관되게 수행할 수 있습니다.

### 고려된 대안 (Alternatives Considered)
- **React Query (TanStack Query):** 강력한 서버 상태 관리 도구이지만, 현재 프로젝트의 규모에서는 별도의 라이브러리 도입 없이 순수 서비스 함수만으로도 충분하다고 판단했습니다.
- **Axios:** `fetch` 대신 사용할 수 있는 훌륭한 라이브러리지만, 현재 브라우저 내장 `fetch`로도 요구사항을 모두 충족할 수 있어 추가 번들 크기 증가를 피하기 위해 채택하지 않았습니다.
